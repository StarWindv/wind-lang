## 语言的风格、类型与变量系统

### I. 类型

#### 1.1.1 语言自身的类型
 - 从高层来看, 它应该是编译型, 所有代码都需要经过编译成为字节码之后再投入运行
 - 从底层来看, 短期内它将作为一门解释型语言存在, 并在后期逐步绑定到机器码做 Native AOT

#### 1.1.2 变量系统
 - 运行时可选的保留类型信息, 仅在启用特定标志位或相关库、函数时才会保留, 否则进行擦除
 - 一定要做编译期求值, 一定!
 - StackFrame 里不存变量表, 我们采用一种完全特殊的名称修饰冒泡来解决变量冲突污染问题, 将所有的变量都简化为一次查表
 - 对于定义时带有类型表达式的变量, 其类型无法更改; 对于未声明类型的变量, 它的类型可以替换, 包括但不仅限于值转型(如 8 -> 8.0)和彻底替换(如 a: Dog -> b: Car)

##### 1.1.5.2 关于名称修饰

栈帧里不存档局部变量表, 变量在编译期按顺序分为固定的几个组纳入HashMap, 

使用名称修饰解决作用域冲突问题, 每个作用域有着自己在当前项目里不冲突的唯一索引, 

举个例子, 
```
fn main() {
    a = 1;
    b = 1;
    output(a, b);
    {
        a = 2;
        output(a);
    }
}
```
在这里我们只有一个函数main, 它自身所在的位置称为顶级作用域, 也就是idx=1, 1_global, 通常这里的变量应该只有 const

而main自己的作用域称为2_main, 第一个a就是2_main_1_a:1, 

而花括号里的那个a在修饰后就是2_main_2_a:1, 因为它处于main的子级作用域所以称为_2, 

:1表示这是此作用域的a的第一次赋值, 而"2_main_2_a:1"表示花括号是2_main的子级作用域, 

这样可以直接区分变量, 然后在各种output后直接drop, 以此类推, 

这样在真正的runtime留下来的只有一个查树再查表的过程, 没有gc, 没有手动生命周期

```
Global = 1
 |—— main = 2
    `main_top` = 2_1
       |—— a -> 2_main_1_a:1
       |—— b -> 2_main_1_b:1
       |—— `Braces` = 2_2
            |—— a -> 2_main_2_a:1
```

##### 1.1.5.3 追加:名称修饰

对于 debug, 名称修饰是一种完全不友好的操作, 它保留下来的只有一个模棱两可的编号和赋值顺序, 
但是我们在编译时是可以得到每个修饰产生的时刻与位置的, 这可以配合 stacktrace 来做 sourcemap 和 debugging

### II. struct/type/group/extra/which/@/::/箭头表达式

#### 2.2.1 概貌
此处的方括号对仅做位置提示, 尖括号对代表可选功能
```wind
struct MyStruct {
    [permission] [name]: [type] <operation>;
}
```

#### 2.2.2 具体示例
```wind
type Email = string where {
    self.length >= 5 && self.length < 20;
    self.matches("@[a-zA-Z]+\\.[a-zA-Z]+");
}
type Name  = string where {
    self.length > 1 && self.length < 10;
}
struct User {
    pub email: Email;
    pub name : Name;
    pub uid  : int where { uid.length == 11 };
    pub age  : int -> { age > 0 && age < 65 };
}

```

或者这样, 把验证移动到外面
当`group:Identifier`之后跟随了一个类,
那么此处的`self`将自动表示为对应类的实例概念
```wind
group Dispenser: User {
    self.name  -> Name;
    self.email -> Email;
```

### III. Closure

这是一个比较特殊的情况, 理论上来说我们应该实现它, 而且闭包确实在某些应用里很好用

至于实现方法那依旧是多了去了, 最简单的就是多开辟一张表专门用来存放闭包中的变量, 只需要闭包运行时自身自行修改+读取即可
